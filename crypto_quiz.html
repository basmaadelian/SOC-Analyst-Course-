Cryptography for SOC Analysts

Every alert you triage, every encrypted C2 channel you investigate, every certificate anomaly you flag — cryptography is underneath it. This quiz tests your understanding of the core concepts: ciphers, keys, symmetric and asymmetric encryption, PKI, digital signatures, and where each one shows up in your SOC workflow.

SECTION 01 Ciphers, Algorithms & Keys Q1–Q5
Score –/5
Q1
CONCEPT

What is a cipher?

✓ CORRECT
A cipher is the algorithm — the set of mathematical rules — that transforms plaintext into ciphertext (encryption) and back again (decryption). The key is a separate input that drives the cipher. Encoding (option A) and hashing (option D) are different operations entirely.
In your alerts, you'll see cipher names like AES-256-GCM or RC4 in TLS negotiation logs. Weak or deprecated ciphers (RC4, DES) in TLS handshakes are a detection signal — they indicate either a legacy system or an attacker downgrading the connection.
Q2
CONCEPT

What is the role of a key in an encryption algorithm?

✓ CORRECT
The key is the secret input that drives the cipher's transformation. The same plaintext encrypted with AES using two different keys produces completely different ciphertext. This is why key management is the hardest problem in cryptography — if you lose control of the key, the algorithm itself gives you nothing.
Ransomware generates a unique AES key per victim, encrypts files, then sends the key to the attacker's C2 server. The victim can't recover files without that key — which is why C2 communications during the encryption phase are a critical detection window.
Q3
APPLICATION

AES-256 means AES is used with a 256-bit key. What does increasing the key length primarily affect?

✓ CORRECT
A 256-bit key has 2²⁵⁶ possible values — a number larger than the estimated atoms in the observable universe. Doubling the key length squares the brute-force difficulty. AES-128 and AES-256 use the same cipher with the same structure; only the key size and number of processing rounds differ.
When reviewing TLS configurations on CorpNet's HR portal, AES-128 is acceptable, but seeing AES-56 or DES in negotiated cipher suites should trigger an immediate finding — those are brute-forceable with modern hardware.
Q4
CONCEPT

What is the difference between a block cipher and a stream cipher?

✓ CORRECT
AES is a block cipher — it processes 128-bit blocks at a time. RC4 is a stream cipher — it generates a continuous keystream XORed with each byte. Both can be symmetric. The distinction matters for performance and security properties, not for what data they protect.
RC4 was used in early WEP Wi-Fi encryption and older SSL versions. Its stream cipher implementation had serious weaknesses — statistical patterns in the keystream allowed attackers to recover plaintexts. Seeing RC4 in a TLS negotiation log today is a major red flag.
Q5
SCENARIO

An analyst reviewing CorpNet's firewall logs notices HTTPS traffic to an external IP using TLS 1.0 with cipher suite RC4-MD5. Why is this a finding worth escalating?

✓ CORRECT
TLS 1.0 was deprecated in 2020 (RFC 8996) and contains known vulnerabilities like POODLE and BEAST. RC4 has been prohibited in TLS since RFC 7465. A connection negotiating both suggests either a very old unpatched system or a deliberate downgrade attack — both require investigation.
Modern SOC playbooks flag any TLS version below 1.2 as an automatic finding. If CorpNet's HR portal is accepting TLS 1.0 connections, that's a remediation ticket. If an endpoint is initiating them to an external IP, that's a potential indicator of C2 communication over a deliberately weakened channel.
SECTION 02 Symmetric Encryption & Its Risks Q6–Q11
Score –/6
Q6
CONCEPT

What defines symmetric encryption?

✓ CORRECT
Symmetric encryption uses a single shared secret key for both operations. This makes it fast and efficient for large data volumes — which is why AES is used to encrypt hard drives, backups, and bulk file transfers. The challenge isn't the encryption itself; it's getting that key to the other party securely.
WannaCry used AES-128 symmetric encryption to lock victim files. The AES key was then encrypted with an RSA public key and sent to the attacker's C2 — that's the combination of symmetric and asymmetric encryption in a real-world attack.
Q7
CONCEPT

What is the key distribution problem in symmetric encryption?

✓ CORRECT
If Alice wants to send Bob an AES-encrypted file, she needs Bob to have the key first. But if she sends the key over the internet unprotected, an eavesdropper gets everything. This is the fundamental problem that asymmetric encryption was designed to solve — and why TLS uses RSA or ECDH to exchange a symmetric session key securely.
This is directly relevant to ransomware incident response. Victims can't decrypt their own files because they don't hold the AES key — the attacker does. There is no "symmetric key recovery" without the attacker's cooperation or a flaw in their implementation.
Q8
APPLICATION

CorpNet uses AES-256 to encrypt its file server backups. The encryption key is stored in a text file on the same file server. What is wrong with this setup?

✓ CORRECT
Collocating the key and the encrypted data defeats the entire purpose of encryption. An attacker who compromises the file server gets both — the encrypted backups and the key to decrypt them. Keys must be stored separately, ideally in a dedicated key management system (KMS) or hardware security module (HSM).
In post-breach forensics, this is a critical finding. If CorpNet's incident report shows the attacker accessed both the backup directory and a file named "backup_key.txt" on the same server, the backup data must be treated as fully compromised regardless of the encryption algorithm used.
Q9
CONCEPT

Which of the following is a known risk of symmetric encryption that asymmetric encryption does NOT share?

✓ CORRECT
Symmetric encryption's core risk is key distribution — both parties need the same secret, but sharing it is the exposure point. Asymmetric encryption solves this: you share your public key openly, and only your private key decrypts. Note: option D (non-repudiation) is also a real limitation of symmetric encryption, but it's not unique to it vs. asymmetric — digital signatures handle non-repudiation separately.
TLS uses both: asymmetric cryptography (RSA/ECDH) during the handshake to safely exchange a symmetric session key, then switches to AES for the actual data transfer. This hybrid approach gets the security benefits of asymmetric with the speed of symmetric.
Q10
SCENARIO

A threat intel report states that a ransomware group generates a unique AES key per victim, uses it to encrypt all files, then encrypts the AES key with the attacker's RSA public key. Why is the AES key encrypted with RSA before exfiltration?

✓ CORRECT
This is a classic hybrid encryption pattern used offensively. AES encrypts fast (terabytes of files). RSA protects the AES key so only the attacker — holding the RSA private key — can recover it. The victim can see the encrypted AES key but cannot use it without the RSA private key. This design makes decryption impossible without paying (or finding a flaw in the implementation).
The detection opportunity is during the C2 callback when the encrypted AES key is exfiltrated. If your SIEM or EDR captures that outbound connection before it succeeds, the attacker never receives the key — and technically, neither did the victim. Some ransomware variants fail silently when C2 is blocked, leaving files unencrypted.
Q11
APPLICATION

Why is AES preferred over DES for modern encryption?

✓ CORRECT
DES uses a 56-bit key — meaning 2⁵⁶ possible values. In 1999, the EFF cracked a DES key in under 24 hours using dedicated hardware. AES-128 provides 2¹²⁸ combinations — the difference is astronomical. Both are block ciphers; key size is the decisive factor here.
Finding DES or 3DES in any active system today is an immediate vulnerability finding. PCI DSS explicitly prohibits 3DES in new implementations as of 2023. If CorpNet's legacy HR portal still negotiates DES cipher suites, that's a high-severity finding in any security assessment.
SECTION 03 Asymmetric Encryption Q12–Q16
Score –/5
Q12
CONCEPT

In asymmetric encryption, if Alice wants to send Bob an encrypted message that only Bob can read, which key does Alice use to encrypt it?

✓ CORRECT
Alice encrypts with Bob's public key. Only Bob's matching private key can decrypt it. Bob's public key is freely distributed — anyone can use it to send Bob an encrypted message — but only Bob holds the private key needed to read it. This solves the key distribution problem completely.
This is exactly how encrypted email works (PGP/S/MIME). It's also the foundation of TLS: your browser uses the web server's public key (from its certificate) during the handshake to securely negotiate the session key. No prior secret exchange needed.
Q13
APPLICATION

Why is asymmetric encryption NOT used to encrypt large files directly, despite its stronger security model?

✓ CORRECT
RSA and ECC are orders of magnitude slower than AES for bulk data. The practical solution is hybrid encryption: use asymmetric to securely exchange a symmetric key, then use that symmetric key for the actual data. This is exactly what TLS, PGP, and ransomware all do.
When you see a TLS handshake in Wireshark, the "Client Key Exchange" or ECDH parameters in the handshake phase are the asymmetric part. After that, the stream switches to symmetric AES for all application data. The handshake is expensive; the data transfer is cheap.
Q14
CONCEPT

What is the mathematical relationship between a public key and its corresponding private key?

✓ CORRECT
The key pair is generated together from a mathematical relationship (prime factorization in RSA, elliptic curve operations in ECC). The public key is derived from the private key, but reversing that derivation — computing the private key from the public key — requires solving a computationally intractable problem. This one-way relationship is the foundation of asymmetric security.
Private key compromise is catastrophic. If an attacker extracts CorpNet's web server private key, they can impersonate the server, decrypt past recorded TLS sessions (if forward secrecy wasn't used), and sign anything as the server. Certificate revocation and reissuance is mandatory upon private key exposure.
Q15
SCENARIO

CorpNet's security team discovers an attacker performed a man-in-the-middle attack on the HR portal, intercepting TLS traffic. The attacker presented a fake certificate for corpnet.com. What cryptographic mechanism should have prevented the browser from trusting that certificate?

✓ CORRECT
A browser trusts a certificate because a trusted CA has cryptographically signed it. An attacker can generate a certificate for any domain, but they cannot forge the CA's digital signature without the CA's private key. Without a valid CA signature, the browser displays a certificate error — which is exactly the warning users should never click through.
Corporate SSL inspection proxies do this deliberately — they intercept TLS, present their own certificate signed by an internal corporate CA (which is installed as trusted on all endpoints). This is legitimate and is how your NGFW inspects encrypted traffic. But it means your SIEM will see the proxy's certificate, not the destination site's.
Q16
SOC LENS

An EDR alert fires on a CorpNet endpoint: a process generated an RSA key pair and immediately made an outbound connection. What is the most likely malicious interpretation?

✓ CORRECT
Ransomware commonly generates its key pair locally, sends the public key to the attacker's C2 (keeping the private key off-victim), then uses the public key to encrypt the symmetric file encryption key. This sequence — key generation followed immediately by outbound connection — from a non-administrative process on an endpoint is a high-confidence ransomware pre-encryption indicator.
This is a kill-chain moment. If EDR fires on this behavior, the correct response is to isolate the endpoint immediately — before file encryption begins. Every minute of delay is potentially thousands of encrypted files. The outbound connection attempt is your detection window.
SECTION 04 Certificate Authorities & PKI Q17–Q21
Score –/5
Q17
CONCEPT

What problem does a Certificate Authority (CA) solve in asymmetric encryption?

✓ CORRECT
The trust problem: anyone can generate an RSA key pair and claim it belongs to "bank.com." A CA solves this by verifying identity (domain ownership, organization identity, extended validation) and signing the certificate with its own private key. Browsers trust specific root CAs — if a certificate chains to a trusted root, the browser accepts the identity claim.
There are roughly 150 trusted root CAs in a typical browser trust store. Any of them can sign a certificate for any domain. This is why rogue or compromised CAs are catastrophic — DigiNotar issued fraudulent Google certificates in 2011, enabling nation-state MITM attacks on Iranian users. DigiNotar was revoked and went bankrupt within weeks.
Q18
CONCEPT

What is a certificate chain of trust?

✓ CORRECT
Root CAs sign intermediate CAs; intermediate CAs sign end-entity (leaf) certificates. The browser validates the entire chain: leaf → intermediate → root. Root CA private keys are kept offline in hardware security modules — they're too valuable to expose. Intermediates do the day-to-day signing.
When you see a certificate error "certificate chain incomplete," the server isn't sending its intermediate CA certificate. Browsers can't validate the chain and display a warning. This is a misconfiguration, not an attack — but it looks identical to a self-signed certificate warning to an untrained user. Know the difference when investigating user-reported phishing.
Q19
APPLICATION

What is certificate revocation, and why does it matter to a SOC analyst?

✓ CORRECT
If a private key is stolen, the certificate is still technically valid until expiry — which could be years. Revocation (via CRL or OCSP) tells browsers to stop trusting it immediately. But revocation checking is notoriously unreliable — many browsers soft-fail if they can't reach the OCSP responder.
In an incident involving a compromised web server, determining whether the TLS certificate's private key was accessed is critical. If yes, revocation and certificate reissuance are mandatory response steps. Certificate Transparency logs (crt.sh) let you verify what certificates have been issued for your domain — unexpected entries may indicate an attacker obtained a fraudulent cert.
Q20
SCENARIO

CorpNet's SOC receives an alert: a user's browser accepted an HTTPS certificate for hr.corpnet.com signed by an internal CA not in the corporate trust store. What should the analyst investigate first?

✓ CORRECT
A rogue root CA being trusted by a browser is a serious finding. Malware, corporate spyware, and MITM attack tools (like mitmproxy) install their own CA into the system trust store to intercept HTTPS without browser warnings. If the CA isn't in your standard corporate baseline, that endpoint needs immediate investigation.
Enumerate trusted CAs on endpoints as part of your baseline. Group Policy or MDM should enforce the approved CA list. Any deviation — an extra root CA not in the approved list — is an immediate escalation. Check the endpoint for evidence of SSL inspection tools, proxy configurations, and malware simultaneously.
Q21
SOC LENS

What does Certificate Transparency (CT) logging provide that helps threat hunters?

✓ CORRECT
Every certificate issued by a trusted CA must be logged in public CT logs. Tools like crt.sh let you search all certificates ever issued for a domain. This is powerful for defenders: if an attacker obtains a fraudulent certificate for hr.corpnet.com from a rogue CA, it will appear in CT logs — often before any attack begins.
Threat hunters use crt.sh to monitor certificate issuance for their domains. New subdomains appearing in CT logs (like "vpn2.corpnet.com" that your team didn't create) can indicate attacker reconnaissance or phishing infrastructure setup. This is a proactive detection technique, not a reactive one.
SECTION 05 Digital Signatures Q22–Q25
Score –/4
Q22
CONCEPT

When Alice digitally signs a document, which key does she use to create the signature?

✓ CORRECT
Signing uses the sender's private key. Only Alice holds her private key — so only she could have produced that signature. Anyone with Alice's public key can verify it. This is the inverse of encryption: encrypt with recipient's public key; sign with sender's private key.
Code signing works on this principle. Windows checks that executables are signed by a trusted publisher before running them. Malware that uses stolen or forged code signing certificates (like the SolarWinds attackers did) bypasses this control — which is why signed-but-unknown binaries still warrant investigation.
Q23
CONCEPT

A digital signature provides two security properties. Which pair is correct?

✓ CORRECT
A digital signature proves: (1) this specific person signed it — because only they hold the private key, and (2) the content is unchanged — because the signature is computed over a hash of the document; any modification breaks the hash and invalidates the signature. Combined, this gives non-repudiation: the signer cannot later deny having signed it.
Email DKIM (DomainKeys Identified Mail) uses digital signatures. The sending mail server signs each email with the domain's private key; receiving servers verify with the public key in DNS. When you're investigating a phishing email, checking whether DKIM passed tells you if it genuinely came from the claimed sending domain — or if the from address was spoofed.
Q24
APPLICATION

Why does a digital signature use a hash of the document rather than signing the entire document directly?

✓ CORRECT
RSA signing a 1GB file directly would be prohibitively slow. Instead: hash the document (SHA-256 produces 32 bytes regardless of input size), then sign the 32-byte hash with RSA. Since any change to the document changes the hash, integrity is preserved. The signature covers the hash; the hash covers the document.
File integrity monitoring (FIM) in your SIEM uses this principle — it hashes critical system files at baseline and alerts when hashes change. If an attacker modifies a system binary, the hash changes and the alert fires, even if the file size, timestamps, and name are identical. This is one of the hardest controls for attackers to evade silently.
Q25
SCENARIO

CorpNet's HR team receives a digitally signed email from "CEO@corpnet.com" authorizing an urgent wire transfer. The DKIM signature passes, but the email came from an external server. What does this tell the analyst?

✓ CORRECT
DKIM proves the email was signed by a key associated with the domain — not that the CEO personally sent it. If the CEO's account was compromised (BEC — Business Email Compromise), or if the DKIM private key was stolen, an attacker can send DKIM-valid emails as any address at that domain. Authentication passes; authorization still requires human verification for high-value requests like wire transfers.
BEC is the highest-revenue cybercrime category by financial loss. The playbook is: compromise executive email → send urgent financial requests → DKIM/SPF/DMARC all pass because the account is legitimate. The defense isn't cryptographic — it's procedural: out-of-band verification (phone call to a known number) for any financial authorization. A passing digital signature is necessary but not sufficient evidence of legitimacy.
